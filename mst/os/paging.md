# Non-Contiguous Memory Allocation and Paging

---

### Core Concepts

- **Non-Contiguous Memory Allocation**:  
  Unlike contiguous allocation where a process must occupy a single continuous block of memory, non-contiguous allocation allows a process to be divided and placed in multiple separate locations in RAM.  
  - Example: A 6KB process can be split into three 2KB partitions and placed in three separate holes of 2KB each.
  - This division helps to **remove external fragmentation**, a problem where free memory is split into small, unusable chunks.

- **External Fragmentation**:  
  Occurs in contiguous allocation when free memory is broken into small holes that are too small to satisfy incoming process requests. Non-contiguous allocation eliminates this by splitting processes.

- **Challenges in Non-Contiguous Allocation**:  
  - The holes in memory are created dynamically and change size and number at runtime, making the process of dividing and allocating memory **time-consuming and costly**.
  - Each time a process arrives, the system must check available holes, their sizes, and divide the process accordingly, which wastes time.

---

### Paging: The Practical Solution

- To reduce the overhead of dividing processes dynamically, **paging** is introduced.
- **Paging** involves dividing both the process and memory into fixed-size blocks **before** loading the process into RAM.

| Term          | Definition                                        |
|---------------|-------------------------------------------------|
| Page          | Fixed-size partition of a process in secondary memory (e.g., disk). |
| Frame         | Fixed-size block of main memory (RAM) where pages are loaded.       |

- **Key Principle**:  
  The size of a **page** is always equal to the size of a **frame**. This ensures pages fit perfectly into frames without wasted space.

- Pages are created in the secondary memory (hard disk) before the process arrives in RAM. Similarly, RAM is also partitioned into frames beforehand.

- During allocation, pages of a process can be loaded into any free frames in memory, regardless of their physical locations, enabling **non-contiguous placement** without fragmentation.

---

### Example and Explanation

- Suppose RAM size = 8KB, frame size = 1KB → Number of frames = 8 (frames 0 to 7).
- A process P1 of size 2KB will be divided into 2 pages of 1KB each.
- Similarly, other processes (P2, P3, P4) of 2KB each are also divided into pages.
- If enough contiguous frames are free, pages can be placed contiguously; if not, pages are placed non-contiguously.
- When processes finish and leave memory, holes (free frames) of 1KB each are created.
- A new process of size 4KB will be divided into 4 pages and can be allocated to any 4 available frames, even if those frames are scattered, thus **eliminating external fragmentation**.

---

### Key Insights

- **Non-contiguous allocation removes external fragmentation** by allowing process parts to be placed separately.
- Dynamic division of processes at runtime is costly and inefficient.
- **Paging solves this by pre-dividing processes into pages** and memory into frames of equal size.
- This pre-division allows faster memory allocation and better utilization.
- The fixed size equality of pages and frames is essential to ensure proper fit and efficient memory use.
- Paging enables memory allocation in a flexible, non-contiguous manner without fragmentation issues.

---

### Conclusion

- Non-contiguous memory allocation improves memory utilization by splitting processes across multiple memory locations.
- However, its runtime cost is high due to dynamic process division and hole management.
- Paging introduces a structured, efficient approach by pre-dividing processes and memory into pages and frames of equal size.
- This technique allows flexible, efficient, and fragmentation-free allocation of memory to processes.
- **Paging is a fundamental solution to the problems of contiguous memory allocation and external fragmentation.**

# Paging and Address Translation in Memory Management

---

### Key Concepts and Definitions

| Term                     | Definition                                                                                  |
|--------------------------|---------------------------------------------------------------------------------------------|
| **Page**                 | A fixed-size block of a process in virtual memory, here assumed as 2 bytes per page.        |
| **Frame**                | A fixed-size block of physical memory (main memory), same size as a page (2 bytes here).    |
| **Process size**          | Total size of the process, e.g., 4 bytes for process P1.                                    |
| **Logical Address**       | Address generated by the CPU to access a byte in the process, consists of page number and offset within the page. |
| **Physical Address**      | Actual location in main memory where the byte is stored, consists of frame number and offset within the frame.      |
| **Page Table**            | Data structure used to map pages of a process to frames in main memory.                     |
| **Memory Management Unit (MMU)** | Hardware component that performs the address translation from logical to physical addresses using the page table. |

---

### Paging Example Breakdown

- Process $P1$ has a **size of 4 bytes**.
- Page size = 2 bytes.
- Number of pages in $P1 = \frac{4}{2} = 2$ pages (Page 0 and Page 1).
- Main memory size = 16 bytes.
- Frame size = page size = 2 bytes.
- Number of frames in main memory = $\frac{16}{2} = 8$ frames, numbered 0 through 7.

---

### Memory Layout and Placement

- $P1$’s Page 0 is loaded into **Frame 2** (bytes 4 and 5 of memory).
- $P1$’s Page 1 is loaded into **Frame 4** (bytes 8 and 9 of memory).
- CPU requests a byte by logical address (e.g., byte 3 of $P1$).
- The **logical address 3** corresponds to:
  - Page number = 1 (since pages are 2 bytes each: bytes 0-1 in Page 0, bytes 2-3 in Page 1).
  - Offset within the page = 1 (second byte in Page 1).

---

### Address Translation Process

1. **CPU generates a logical address** (e.g., 3).
2. Logical address consists of:
   - **Page number**: For byte 3, page number = 1.
   - **Page offset**: For byte 3, offset = 1.
3. MMU uses the page number to look up the **page table entry**.
   - Page 0 → Frame 2
   - Page 1 → Frame 4
4. MMU replaces the page number with the corresponding **frame number**.
5. Physical address is constructed by concatenating the frame number and the offset.
   - Frame 4 = binary 100 (3 bits, since 8 frames).
   - Offset = 1 bit (to represent 2 bytes per frame).
6. The logical address 3 is translated to physical address 9 (byte 9 in main memory).

---

### Bit Representation and Address Size

| Parameter                     | Value/Details                                   |
|------------------------------|------------------------------------------------|
| Number of pages (for $P1$)    | 2 (Page 0 and Page 1)                           |
| Bits to represent pages       | 1 bit (0 or 1)                                  |
| Number of frames              | 8                                              |
| Bits for frame number         | 3 bits (to represent 0 to 7)                    |
| Page/frame size              | 2 bytes                                         |
| Bits for offset within page/frame | 1 bit (to represent byte 0 or 1)              |
| Total bits for physical address | 4 bits (frame number bits + offset bits)      |

---

### Logical vs Physical Address Example

| Logical Address (Byte) | Page Number | Page Offset | Frame Number (from page table) | Physical Address (Frame Number + Offset) |
|-----------------------|-------------|-------------|-------------------------------|------------------------------------------|
| 3                     | 1           | 1           | 4                             | 9                                        |
| 1                     | 0           | 1           | 2                             | 5                                        |

---

### Important Notes

- **Page size and frame size must be equal** to allow straightforward placement of pages into frames.
- The **CPU generates logical addresses** unaware of the paging mechanism.
- The **MMU and page table perform the required mapping** to physical addresses.
- Logical address translation involves:
  - Extracting the page number and offset.
  - Using the page number to find the corresponding frame number.
  - Combining frame number with offset to form the physical address.
- Memory is **byte addressable**; hence addresses point to bytes, not larger units.
- Address counting and page/frame numbering start from 0 (zero-based indexing).
- The **bit-length of page offset and frame offset must be identical** to maintain the correct byte position within pages and frames.

---

### Summary of Process Flow

- CPU requests a byte by generating a logical address.
- Logical address is decomposed into page number and offset.
- MMU consults the page table to find the frame number corresponding to the page number.
- Physical address is constructed by concatenating frame number and offset bits.
- Data is fetched from physical memory using the physical address and sent to the CPU.

# Page Replacement Algorithm and FIFO 

The video explains the concept of **page replacement algorithms** in the context of **virtual memory management**. The key idea is that the entire process cannot be loaded into the limited main memory at once. Instead, only a subset of pages that are currently needed are brought into memory, and when memory is full, some pages must be **replaced (swapped out)** to accommodate new pages.

---

### Core Concepts

- **Virtual Memory**: Allows multiple processes to appear as if they have full memory access, but only loads pages that are demanded.
- **Page Replacement**: When main memory (frames) is full and a new page is needed, one existing page must be removed to free space.
- **Page Fault**: Occurs when a demanded page is not found in main memory, requiring it to be fetched from slower secondary storage (e.g., hard disk).
- **Page Hit**: When the demanded page is already in main memory.

---

### Common Page Replacement Algorithms

1. **First-In-First-Out (FIFO)**
2. **Optimal Page Replacement** (*Not specified in detail*)
3. **Least Recently Used (LRU)** (*Not specified in detail*)

This video focuses exclusively on the **FIFO** algorithm.

---

### FIFO Algorithm Explanation

- Maintains pages in memory in the order they were loaded.
- When a new page needs to be loaded and memory is full, the **oldest page** (first loaded) is removed.
- Replacement happens in a cyclic, frame-wise manner.

---

### Example Setup

- **Reference string (page requests):** $7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 1, 2, 0$
- **Number of frames:** 3 (fixed for this example)
- CPU demands pages in sequence; each demand is checked against current frames.

---

### FIFO Page Replacement Process (Key Stages)

| Step | Page Requested | Frame Status Before | Action         | Page Fault / Hit | Frame Status After       |
|-------|----------------|---------------------|----------------|------------------|-------------------------|
| 1     | 7              | Empty               | Load 7 into Frame 1 | Fault            | [7, -, -]               |
| 2     | 0              | [7, -, -]           | Load 0 into Frame 2 | Fault            | [7, 0, -]               |
| 3     | 1              | [7, 0, -]           | Load 1 into Frame 3 | Fault            | [7, 0, 1]               |
| 4     | 2              | [7, 0, 1]           | Replace oldest (7) with 2 | Fault       | [2, 0, 1]               |
| 5     | 0              | [2, 0, 1]           | Found in frame 2   | Hit              | [2, 0, 1]               |
| 6     | 3              | [2, 0, 1]           | Replace next oldest (0) with 3 | Fault    | [2, 3, 1]               |
| 7     | 0              | [2, 3, 1]           | Replace next oldest (1) with 0 | Fault    | [2, 3, 0]               |
| 8     | 4              | [2, 3, 0]           | Replace oldest (2) with 4 | Fault       | [4, 3, 0]               |
| 9     | 2              | [4, 3, 0]           | Replace next oldest (3) with 2 | Fault    | [4, 2, 0]               |
| 10    | 3              | [4, 2, 0]           | Replace next oldest (0) with 3 | Fault    | [4, 2, 3]               |
| 11    | 0              | [4, 2, 3]           | Replace oldest (4) with 0 | Fault       | [0, 2, 3]               |
| 12    | 3              | [0, 2, 3]           | Found in frame 3   | Hit              | [0, 2, 3]               |
| 13    | 1              | [0, 2, 3]           | Replace next oldest (2) with 1 | Fault    | [0, 1, 3]               |
| 14    | 2              | [0, 1, 3]           | Replace next oldest (3) with 2 | Fault    | [0, 1, 2]               |
| 15    | 0              | [0, 1, 2]           | Found in frame 1   | Hit              | [0, 1, 2]               |

---

### Summary of Results

| Metric                   | Value           |
|--------------------------|-----------------|
| Total number of page references | 15              |
| Number of page faults (misses)  | 12              |
| Number of page hits              | 3               |
| Hit ratio                       | $$\frac{3}{15} \times 100 = 20\%$$ |
| Miss ratio (page fault ratio)   | $$\frac{12}{15} \times 100 = 80\%$$ |

---

### Key Insights

- **Page fault occurs whenever the page requested is not in any frame.**
- **Page replacement only happens when all frames are full.**
- FIFO replacement is simple: remove pages in the order they entered memory.
- **High page fault rate (80%) in this example illustrates FIFO’s limitation** in some cases.
- Understanding page hit and fault ratios is critical to measuring algorithm efficiency.
- Minimizing page faults is important because accessing the hard disk (on page faults) is slow and degrades system performance.

#  Thrashing in Operating Systems

Thrashing is a critical performance issue in operating systems closely related to the **degree of multi-programming**, which refers to the number of processes loaded into RAM simultaneously. This concept is central to achieving high CPU utilization but also brings about challenges when managing limited RAM resources.

---

### Core Concepts

- **Degree of Multi-Programming:** The number of processes resident in RAM at any time.
- Increasing this degree generally **improves CPU utilization** because when some processes perform I/O operations and become blocked, other ready processes can use the CPU, preventing it from idling.
- However, **RAM size is limited**, so entire processes cannot always fit into RAM simultaneously.

---

### Paging and Its Role

- To handle limited RAM, systems use **paging**, where processes are divided into smaller units called **pages**.
- Instead of loading complete processes, only **one or a few pages** of each process are loaded into RAM.
- This strategy allows many processes to be partially present in RAM, maximizing the degree of multi-programming.
- For example, with 100 processes, loading one page from each allows all processes to appear "active" in RAM without fully loading any single process.

---

### Page Faults and Their Impact

- A **page fault** occurs when the CPU requests a page not currently in RAM.
- Handling a page fault requires fetching the page from the slower hard disk into RAM, which takes significant time (known as **page fault service time**).
- Excessive page faults cause the operating system to spend most of its time servicing these faults instead of executing processes, leading to performance degradation.
- This state is called **thrashing**.

---

### Relationship Between Degree of Multi-Programming and CPU Utilization

| Degree of Multi-Programming | CPU Utilization Behavior                    |
|-----------------------------|---------------------------------------------|
| Low                         | CPU utilization is low due to idle time when processes wait for I/O. |
| Increasing                  | CPU utilization rises as more processes keep CPU busy.               |
| Optimal Point ($\lambda$)    | Maximum CPU utilization achieved.                                    |
| Beyond $\lambda$             | CPU utilization drops sharply due to thrashing (excessive page faults). |

- Initially, increasing the degree of multi-programming improves CPU utilization.
- After a critical threshold $\lambda$, further increasing processes causes **thrashing** and CPU utilization plummets.

---

### Causes of Thrashing

- Too many processes loaded simultaneously.
- CPU requests pages not in RAM, causing frequent page faults.
- Excessive time spent swapping pages in and out of RAM, leaving little CPU time for actual computations.

---

### Solutions to Thrashing

- **Increase RAM size:** More memory reduces page faults by accommodating more pages. However, this is often impractical to do frequently.
- **Control degree of multi-programming:** Managed by the **long-term scheduler**, which controls how many processes are admitted into RAM.
  - Decreasing the rate at which new processes are loaded or limiting the total number of concurrent processes prevents thrashing.
- These control mechanisms balance utilization and prevent the system from entering thrashing conditions.

---

### Key Insights

- **Thrashing occurs when the system tries to maximize process concurrency beyond RAM capacity, leading to excessive page faults.**
- **The long-term scheduler plays a vital role in preventing thrashing by limiting the degree of multi-programming.**
- Managing the trade-off between CPU utilization and memory limitations is essential for optimal OS performance.
- **Thrashing causes a paradoxical drop in CPU utilization despite more processes being loaded.**

---

### Definitions

| Term                     | Definition                                                                                 |
|--------------------------|--------------------------------------------------------------------------------------------|
| Thrashing                | Excessive paging causing a system to spend more time handling page faults than executing processes. |
| Degree of Multi-Programming | Number of processes loaded into RAM simultaneously.                                         |
| Page Fault               | An event triggered when a requested page is not found in RAM, requiring a fetch from disk.  |
| Page Fault Service Time  | The time taken to load a missing page from disk into RAM.                                   |
| Long-Term Scheduler      | OS component responsible for admitting processes into RAM and controlling multi-programming degree. |

# Virtual Memory Concepts 

Virtual memory is a fundamental concept in modern computing that provides an **illusion to the programmer** that processes larger than the available physical main memory can still be executed efficiently. This concept allows systems to overcome the **finite size of main memory** (RAM), which is typically limited (e.g., 8GB, 16GB), while processes and their memory requirements continue to grow.

### Key Insights and Advantages

- **Illusion of Larger Process Size:** Virtual memory enables execution of processes whose size exceeds the physical memory capacity by only loading necessary parts (pages) of a process into main memory.
- **Increased Degree of Multiprogramming:** By loading only parts of multiple processes simultaneously, virtual memory allows more processes to share the limited main memory, enhancing system utilization and throughput.
- **Paging and Locality of Reference:** Processes are divided into fixed-size pages, which correspond to frames in main memory. Instead of loading an entire process, only the required pages are loaded based on the principle of locality of reference, which assumes that pages near a currently accessed page are likely to be accessed soon.
- **Partial Loading of Multiple Processes:** Main memory contains pages from many processes concurrently, allowing the illusion that all processes are fully resident in memory.

### Working Mechanism of Virtual Memory

- **Page Tables and Memory Management Unit (MMU):** The MMU uses a page table to translate logical addresses generated by the CPU into physical frame addresses in main memory.
- **Page Faults:** When the CPU requests a page not currently in main memory (indicated by an invalid or absent bit in the page table), a **page fault** occurs.
- **Trap and Interrupt:** The page fault generates a trap (a type of interrupt), transitioning control from user mode to the operating system (OS), initiating a context switch.
- **Page Fault Handling Steps:**
  1. OS authenticates the user/request.
  2. OS locates the requested page in the logical address space (usually on the hard disk).
  3. OS brings the page from disk into a free frame in main memory.
  4. Updates the page table to mark the page as present and store the frame number.
  5. Returns control back to the user process to continue execution.

### Page Replacement and Swapping

- **Swap-in and Swap-out (Roll-in and Roll-out):** To manage limited physical memory, pages are swapped in when needed and swapped out to disk when memory is full.
- **Page Replacement Algorithms:** Various algorithms like FIFO (First In First Out), LRU (Least Recently Used), and MRU (Most Recently Used) are used to decide which pages to evict when loading new pages.

### Performance Considerations: Effective Memory Access Time (EMAT)

The **effective memory access time** depends on the probability of page faults and the time taken to service them versus normal memory accesses.

- Let:
  - $P$ = Probability of a page fault occurring.
  - $1 - P$ = Probability of a page hit (no page fault).
  - $MA$ = Main memory access time (in nanoseconds).
  - $S$ = Page fault service time (in milliseconds).

- The formula for **Effective Memory Access Time** is:

  $$
  EMAT = P \times S + (1 - P) \times MA
  $$

- **Typical time scales:**
  - Main memory access time ($MA$) is extremely fast, measured in **nanoseconds**.
  - Page fault service time ($S$) is much slower, measured in **milliseconds** (due to slow hard disk access and context switching).
  - Since 1 millisecond = 1,000,000 nanoseconds, page faults drastically increase memory access time.

### Page Fault Impact and Thrashing

- Excessive page faults cause **thrashing**, where the CPU spends more time handling page faults than executing processes, leading to significant performance degradation.
- Minimizing page faults and increasing page hits improves the effective memory access time and system performance.
